| 技术组件   | 简化方案                            | 容量规划                   |
| :--------- | :---------------------------------- | :------------------------- |
| 应用服务器 | Spring Boot内置Tomcat               | 最大线程数200，队列100     |
| 缓存       | 单机Redis（用于库存扣减、验证码）   | 1G内存，禁用持久化（可丢） |
| 消息队列   | RabbitMQ（或可选：直接线程池+延迟） | 单机版，用于订单超时取消   |
| 数据库     | MySQL 8.0（单库，不做读写分离）     | 表数量<20，索引优化        |
| 文件存储   | 阿里云OSS（商家图片）               | 绑定CDN加速域名            |

### 阶段二：单体成熟化（核心增长）

**架构特征**：**模块化单体**（分包但未拆分部署）+ 读写分离 + 多级缓存
**部署形态**：2台ECS（4C8G）+ 主从RDS + Redis Cluster

| 升级点       | 方案                                                         |
| :----------- | :----------------------------------------------------------- |
| **数据库**   | 主从复制：订单写主库，报表读从库。使用Sharding-JDBC读写分离。 |
| **缓存**     | 引入Caffeine作为一级缓存（商家列表、热门商品），TTL 30秒；Redis为二级缓存。 |
| **消息队列** | RocketMQ正式引入，订单创建→发送Topic给配送、营销服务（仍在同一进程内模块调用，但通过MQ异步解耦，为微服务做准备）。 |
| **定时任务** | XXL-Job部署，执行每日对账、过期拼单清理。                    |
| **搜索**     | 集成Elasticsearch，商品信息同步，实现LBS搜索。               |
| **新功能**   | 拼单贪心匹配（单机内存计算），WebSocket推送骑手位置。        |

**性能目标**：

- 下单TPS达到 **4500+**（通过MQ异步化 + Redis库存）

### 阶段三：微服务架构（规模化商业）

**架构特征**：**按业务边界拆分8个服务**，独立部署、独立数据库
**部署形态**：K8s集群（至少3节点），每个服务2~6副本

| 服务         | 拆分粒度               | 数据库隔离                 | 主要中间件                          |
| :----------- | :--------------------- | :------------------------- | :---------------------------------- |
| **用户服务** | 用户、会员、积分       | user_db（分库分表）        | Redis（session）、Seata TCC         |
| **商家服务** | 门店、商品、审核       | shop_db + ES               | OSS、OCR、Caffeine                  |
| **订单服务** | 订单CRUD、状态机       | order_db（按user_id分片）  | RocketMQ（事务消息）、Seata AT      |
| **调度服务** | 拼单匹配、派单、ETA    | dispatch_db + TiDB（报表） | Redis GEO、ML模型（PyTorch）        |
| **配送服务** | 骑手管理、轨迹         | delivery_db + 时序库       | WebSocket、高德API                  |
| **营销服务** | 优惠券、满减、会员权益 | marketing_db               | 规则引擎Drools                      |
| **通知服务** | 短信、推送             | 无                         | 阿里云SMS、极光推送                 |
| **网关**     | 鉴权、限流、路由       | 无                         | Sentinel（限流规则从Nacos动态加载） |

**关键设计**：

- **服务间通信**：Dubbo（高并发场景如订单→库存） + OpenFeign（低频管理类接口）
- **分布式事务**：支付回调触发TCC（订单服务try锁库存，用户服务try增积分，营销服务try发券）；失败则全部cancel。
- **智能调度**：使用**匈牙利算法**每10秒滑动窗口匹配拼单，目标函数最小化总配送成本。ETA预测基于XGBoost模型（特征：时段、天气、骑手负载、商家历史出餐时间）。
- **高可用**：多机房部署，核心服务双活；Canal同步MySQL binlog到备机房。

**性能目标**：

- 日活50万+，峰值订单QPS 2000+，系统可用性99.99%。

## 2. 核心技术栈选型深度方案

针对即时零售“薛定谔库存”、“动态 ETA 预测”等四大业务痛点，平台核心技术栈选型及落地场景规约如下：

### 2.1 基础运行时与核心中间件选型

#### ① 基础运行环境：Spring Boot 3.x + JDK 21 (虚拟线程机制)

- **选型考量：** 即时零售系统涉及大量骑手高频 LBS 位置上报、WebSocket 长连接推送、后厨视频流转等典型的 I/O 密集型场景。  
- **落地机制：** 全面弃用传统高成本的平台线程池方案，利用 **JDK 21 虚拟线程 (Virtual Threads)** 挂起等待特性，让单个操作系统线程能够调度数万个并发 I/O 任务，大幅提升网关层与配送服务的吞吐性能。  

#### ② 消息队列：RocketMQ (事务消息与延迟队列)

- **选型考量：** 拼单合并涉及复杂的逆向状态流（如拼单失败退款、支付成功增加积分等），需要严格保证跨服务的最终一致性；同时需要处理“30分钟未支付自动取消”的超时控制。  
- **落地机制：**
  - **延迟消息：** 用户下单成功发送一条 30分钟 延迟消息至 RocketMQ，由订单服务监听，超时未支付则执行乐观锁关单并回滚 Redis 库存。  
  - **事务消息：** 在拼单失败自动退款场景中，通过 RocketMQ 事务消息联动本地补偿任务表，确保支付系统退款与订单服务状态变更的绝对原子性。  

#### ③ 分布式事务：Seata (AT 与 TCC 混合模式)

- **选型考量：** “支付成功 $\rightarrow$ 扣减实体库存 $\rightarrow$ 核销优惠券 $\rightarrow$ 增加用户积分”链路短且对实时性一致性要求极高。  
- **落地机制：** 采用 **Seata 混合模式**。对于订单与库存等核心关系库，走 **AT 模式**（无侵入、自动回滚）；对于促销优惠券、积分商城等跨库、跨服务的异步调用，采用 **TCC 模式**（Try-Confirm-Cancel），通过显式接口冻结与确认释放，保证大流量下交易链条的金融级稳定性。  

### 2.2 存储层与大规模高并发检索选型

#### ① 核心关系型数据库：MySQL 8.0 + ShardingSphere + TiDB

- **选型考量：** 阶段三要求支撑日活 50万+，单表订单数据将呈爆发式增长，且需要兼顾商户实时财务对账及大屏 BI 报表的 HTAP 混合负载需求。  
- **落地机制：**
  - **ShardingSphere：** 对核心订单库 `order_db` 采用 **ShardingSphere-JDBC** 进行分库分表，选用 `user_id` 作为分片键（Sharding Key），解决单表并发写入瓶颈。  
  - **TiDB：** 将商户端、运营后台等需要跨库关联的多维统计报表、每日对账流水迁移至 **TiDB** 数据库，利用其 Raft 协议的行存/列存分离架构，在不影响主交易链路的前提下支持 Flink 实时大屏计算与实时报表负载。  

#### ② 分布式缓存与过滤：多级缓存 (Caffeine + Redis Cluster)

- **选型考量：** 防止热点商户、热销单品被瞬时高并发流量冲垮，以及高并发下拼单扣减库存产生超卖。  
- **落地机制：**
  - **销售层 (Redis Cluster)：** 库存数据完全下沉至 Redis，利用 **Lua 脚本**实现原子扣减（`DECR` 校验）。在拼单高并发场景下，通过一条 Lua 脚本批量扣减同一拼单组内所有商品库存，确保操作的原子性。  
  - **本地缓存 (Caffeine)：** 针对商户基本信息、平台系统配置、类目字典等极少变更的数据，采用本地 JVM 级缓存框架 Caffeine，避免网络 I/O 开销，首页商家列表等核心接口 TPS 经多级缓存重构后，单机性能从 300 飙升至 **4500+**。  

#### ③ 搜索引擎：Elasticsearch 8.x (LBS 空间地理检索)

- **选型考量：** 即时零售依赖于用户的即时位置，搜索不仅是文本匹配，更是“地理位置+库存状态+个性化推荐”的多维函数打分。  
- **落地机制：** 店铺坐标采用 ES 的 `geo_point` 结构存储，SKU 数据采用 `nested` 嵌套结构归档。用户执行检索时，通过 ES 层直接过滤当前 3km 范围内且对应前置仓库存 $>0$ 的商品，使用自定义打分脚本 (Function Score) 按距离衰减系数及商品销量复合加权排序，拒绝展现“售罄”造成的无效流量浪费。  

### 2.3 智能调度与大数据流中台选型

#### ① 调度与路径规划：图数据库 + Redis GEO + 高德 Matrix API

- **选型考量：** 骑手智能派单、合单（顺路单匹配）属于运筹优化问题（旅行商问题 TSP），计算密集度极高，必须快速剪枝规避无效路径。  
- **落地机制：**
  - **Redis GEO + 布隆过滤器：** 调度引擎首先利用 `GEORADIUS` 粗筛出当前订单 3km 范围内的常驻骑手，接着利用布隆过滤器快速剔除处于“收工/不接单”状态的骑手，大幅缩减后续算法对不活跃骑手的计算消耗。  
  - **智能合单：** 引入**匈牙利算法 + 时间窗约束**，每 10秒 滚动进行在线匹配。将同一方向、同一小区的订单合并通过高德地图 Matrix API 获取距离矩阵，自研 2-opt 局部搜索路径，动态规避实时拥堵路段，**将单均配送成本直接降低 30%**。  

#### ② 视频流中台：SRS (Simple Realtime Server) + YOLOv8 AI 食安审核

- **选型考量：** 满足商户“后厨直播”增值服务的低延时推流转码，并支持 AI 实时违规告警。  
- **落地机制：** 商家低成本摄像头采集视频，通过 RTMP 协议推流至开源流媒体服务器 **SRS**，SRS 负责进行中转分发，前端使用 **HLS (m3u8)** 进行秒级切片播放。后台启用轻量化 Python 进程引入 **YOLOv8 模型**实时抽帧审核，一旦识别出后厨未戴口罩、鼠患、垃圾桶未盖等食安违规行为，由通知服务实时向平台及商家App推送告警并自动执行下架处罚。
